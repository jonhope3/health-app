## What Is FitTrack?

FitTrack is a **privacy-first, offline-capable health tracking app** built exclusively for the Google Pixel 10 Pro. It combines daily calorie and macronutrient tracking, step counting, and on-device AI-powered food recognition into a single, self-contained application.

### Why Does It Exist?

Most health tracking apps require cloud accounts, sync personal data to remote servers, and rely on internet connectivity. FitTrack takes the opposite approach:

- **Zero cloud dependency** — all data is stored locally on-device via SharedPreferences
- **On-device AI** — Gemini Nano runs directly on the Pixel 10 Pro's TPU/NPU, so food parsing, nutrition label scanning, and coaching tips never leave the phone
- **No accounts or sign-ups** — open the app and start tracking immediately
- **Single-device focus** — optimized specifically for the Pixel 10 Pro hardware and sensor capabilities

### Who Is It For?

Anyone who wants a simple, private, AI-enhanced health tracker without the overhead of subscription services, cloud sync, or social features. It's designed for daily personal use — tracking what you eat, how much you move, and getting lightweight AI-powered insights.

## Core Features

### 1. Home Dashboard
The main screen provides an at-a-glance view of daily progress:
- **Animated Calorie Ring** — circular progress visualization showing calories consumed vs. goal, with an inner ring for step progress
- **AI Coach Tip** — Gemini Nano generates a personalized daily insight based on your food log, step count, and goals (e.g., "You've hit 65% of your protein goal — consider adding a protein-rich snack")
- **Macro Summary** — progress bars for protein, carbs, fat, and sugar against configurable daily targets
- **Quick Actions** — add steps or navigate to food logging directly from the dashboard
- **Celebration Animation** — confetti burst when you reach your calorie goal

### 2. Food Logging (4 Modes)
The app offers four distinct ways to log food:

| Mode | How It Works | AI-Powered? |
|------|-------------|-------------|
| **Search** | Searches the web via DuckDuckGo, then uses Gemini Nano (or regex fallback) to extract nutrition facts from results | ✅ |
| **AI Parse** | Describe what you ate in natural language (e.g., "2 scrambled eggs with toast and butter") and Gemini Nano extracts individual items with estimated calories and macros | ✅ |
| **Scan** | Take a photo or upload an image of a nutrition label; Gemini Nano reads and extracts the values | ✅ |
| **Manual** | Enter food name, calories, and macros directly with autocomplete from previously logged foods | ❌ |

After adding food, entries appear in a **Food Diary** timeline view with edit and delete capabilities. Each entry supports:
- Inline quantity editing with auto-scaling of all macro values
- Unit conversion between grams, ounces, cups, ml, tablespoons, and teaspoons
- Preset quantity buttons for quick selection

### 3. Step Tracking
Steps are tracked through a 3-tier automatic fallback system:
1. **Health Connect** (preferred) — reads from Google Health Connect if available and permissions are granted
2. **Hardware Pedometer** — uses the device's built-in step counter sensor
3. **Manual Entry** — add or set step counts directly

The Steps screen shows:
- Today's step count with progress bar toward goal
- Derived metrics: distance walked (miles), estimated calories burned, active minutes
- 7-day step history with an animated line chart
- Add Steps dialog with "add to total" or "set total" modes

### 4. Settings & Configuration
- **Profile** — set a nickname used in greeting and AI prompts
- **Health Connect Integration** — connect/disconnect with permission management
- **Daily Goals** — calorie target (default 2,000) and step target (default 10,000)
- **Body Measurements** — weight (lbs), height (ft/in), age — used for calorie burn estimates
- **Macro Goals** — daily targets for protein, carbs, fat, and sugar in grams
  - Can be **auto-generated by AI** based on your profile and calorie goal
- **Database Explorer** — developer tool for inspecting raw SharedPreferences data
- **Nuclear Reset** — clear all data and restart fresh

### Key Design Decisions
1. **SharedPreferences over Room** — chosen for simplicity; data is serialized as JSON strings. Trade-off: no transactional safety, no relational queries, limited scalability.
2. **No dependency injection** — ViewModels instantiate their own repositories and services directly. Makes testing harder but reduces boilerplate.
3. **Single-Activity architecture** — one `MainActivity` with a `NavHost` managing four composable destinations via bottom navigation.
4. **Gemini Nano with graceful degradation** — every AI feature has a non-AI fallback path. The app is fully usable without Gemini Nano being available.

### Data Storage
| SharedPreferences File | Key Pattern | Value Type | Purpose |
|----------------------|-------------|------------|---------|
| `fittrack_food` | `food_log_{YYYY-MM-DD}` | JSON `List<DiaryItem>` | Daily food diary entries |
| `fittrack_food` | `custom_foods` | JSON `List<FoodItem>` | Saved food items with usage history |
| `fittrack_goals` | `calorie_goal`, `step_goal`, etc. | Int/Float primitives | User goals and body metrics |
| `fittrack_steps` | `steps_{YYYY-MM-DD}` | Int | Daily step counts |
| `fittrack_meta` | `last_cleanup` | String date | Tracks last data pruning run |

### Data Models
- **DiaryItem** — a single food log entry: id, name, calories, protein, carbs, fat, sugar, timestamp, quantity, meal type
- **FoodItem** — a saved food with nutrition data and usage tracking (count, history, first/last used)
- **NutritionResult** — search or scan result with per-serving data and optional confidence score
- **ParsedFoodItem** — AI-parsed food item with confidence rating
- **MealType** — enum: BREAKFAST, LUNCH, DINNER, SNACK, OTHER (auto-assigned by time of day)

## AI Integration

### Gemini Nano on Pixel 10 Pro
The app uses Google's **Gemini Nano** — a Small Language Model (SLM) running entirely on-device via the ML Kit GenAI (Prompt) API. On the Pixel 10 Pro (Tensor G5), inference is accelerated by the device's NPU/TPU hardware.

**Key hardware characteristics:**
- **Context window:** ~4,096 tokens (input + output combined)
- **Model management:** Handled by Android's AICore system service — the app does not bundle model weights
- **Model download:** After initial device setup, AICore downloads the model over WiFi (can take 1-3 hours)
- **Thermal awareness:** AICore manages thermal throttling; the app should be resilient to temporary inference unavailability

**Initialization flow:**
1. Call `checkStatus()` to verify model availability (`AVAILABLE`, `DOWNLOADABLE`, `DOWNLOADING`, `UNAVAILABLE`)
2. If `DOWNLOADABLE` or `DOWNLOADING`, trigger download via `download()` flow and poll for completion
3. Once available, call `warmup()` to pre-load the model for faster first inference
4. If unavailable after polling, gracefully fall back to non-AI features

### Prompt Engineering Best Practices (Applied)
All prompts in the app follow Google's recommended best practices for on-device SLMs:

| Practice | Implementation |
|----------|---------------|
| **Structured delimiters** | All prompts use `<instruction>`, `<user_data>`, `<examples>`, `<input>` XML-style tags to help the SLM distinguish prompt components from data |
| **Concise few-shot examples** | 1-2 compact examples per prompt to save context window tokens while maintaining output consistency |
| **Explicit output format** | Every prompt specifies "Return ONLY a JSON object/array, no other text or markdown" |
| **Edge case handling** | Label scanning prompt instructs `NOT_FOUND` response for images without nutrition labels |
| **Context-driven behavior** | Coach tip tone is selected deterministically based on user state (not random) |
| **Guideline sections** | Coach prompt includes conditional `<guidelines>` with scenario-specific instructions |

### AI Feature Matrix
| Feature | Service | Prompt Strategy | Fallback |
|---------|---------|----------------|----------|
| **Coach Tips** | `CoachTipUseCase` | Structured prompt with user data, macro goals vs. actual, time-of-day, conditional guidelines | Deterministic rule-based tips (3 variations per scenario) |
| **Food Parsing** | `AiFoodParserService` | Natural language input with structured delimiters and 2 few-shot examples to JSON array | Feature unavailable |
| **Nutrition Lookup** | `AiFoodParserService` | Food name with structured prompt to per-100g JSON nutrition data | Feature unavailable |
| **Label Scanning** | `AiFoodParserService` | Image + structured prompt to JSON with NOT_FOUND sentinel handling | Feature unavailable |
| **Macro Goal Gen** | `SettingsViewModel` | User profile with structured delimiters to recommended daily macro JSON | Manual entry only |

### Nutrition Search Pipeline
Food search uses a **3-tier fallback chain** for maximum reliability:
1. **USDA FoodData Central API** (primary) — Most reliable structured data. Uses `DEMO_KEY` (rate limited). Returns per-100g values.
2. **OpenFoodFacts** (secondary) — Community-maintained product database. Good for branded/packaged foods.
3. **DuckDuckGo web scraping** (tertiary) — Scrapes nutrition facts from search results. Gemini Nano extracts JSON, regex as final fallback.

### AI Reliability Notes
- Gemini Nano is an SLM; responses can be inconsistent for unusual or regional foods
- JSON extraction uses regex to handle SLM quirks (markdown code blocks, extra whitespace)
- All numeric fields use `toDoubleOrNull()` with fallback defaults for resilient parsing
- Confidence scores are self-reported by the model and should be treated as approximate
- The `safeDouble()` helper strips unit suffixes (e.g., "1.5g" to 1.5) from model output
- Coach tip format depends on `**bold**` markers — a log warning fires if the model omits them

### Future AI Improvements
- Set low temperature (0.1-0.3) for data extraction tasks to improve JSON consistency
- Use streaming API for coach tips to improve perceived performance
- Cache AI nutrition lookups to avoid redundant inference
- Add automatic retries for empty or malformed responses before falling back

## Known Limitations

1. **Pixel 10 Pro only** — not tested or designed for other devices; sensor and AI availability may differ
2. **No data backup/export** — if the app is uninstalled, all data is lost
3. **SharedPreferences scaling** — performance may degrade with months of daily food logging; no transactional safety means concurrent writes can cause data loss (Room migration recommended)
4. **USDA API uses `DEMO_KEY`** — rate-limited to 40 requests/hour; not suitable for heavy usage
5. **Web search fallback fragility** — DuckDuckGo scraping (3rd tier fallback) may break if DDG changes their HTML structure; USDA and OpenFoodFacts are preferred
6. **No dark mode** — only a light theme is implemented
7. **No accessibility support** — content descriptions and screen reader optimizations are incomplete
8. **Pedometer resets on app restart** — step counter baseline is lost when the service is recreated (mitigated by Health Connect preference)
9. **No data sync** — single-device only, no cross-device transfer mechanism
